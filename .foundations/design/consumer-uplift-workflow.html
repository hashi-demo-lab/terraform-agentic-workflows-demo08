USE CASE 1 -- CONSUMER MODULE UPLIFT

Automated Module Version Upgrades for HCP Terraform Consumers

A connected GitHub Actions pipeline that detects module version bumps, validates with Terraform, applies a deterministic risk matrix, and auto-merges safe changes -- with @claude interactive escalation for complex upgrades.

DETECT

Dependabot scans the private registry monthly. A fallback cron scanner catches submodule paths Dependabot misses. Both create PRs in the same format.

ASSESS

A deterministic risk matrix classifies changes by semver type, plan resource count, and destroy/replace presence. No AI needed -- purely scripted.

DECIDE

Structured JSON output drives deterministic decisions: auto-merge low-risk patches, request review for medium, block breaking changes. @claude for follow-up.

Escalation

The @claude Loop

When automated analysis labels a PR as needs-review or breaking-change, humans can @claude for interactive follow-up in the same workflow

Low Risk: Auto-Merge

Patch or minor bump, no breaking changes, small plan diff. Pipeline auto-merges with squash. No human intervention needed.

risk:low auto-merge

Medium Risk: @claude Available

Plan changes > 5, or breaking changes that were auto-adapted. Labels applied, reviewer requested. Comment says: @claude for deeper analysis.

risk:medium needs-review

High Risk: @claude Recommended

Major version bump, DESTROY in plan, or security finding. Detailed analysis posted. @claude can make code changes and push to branch.

risk:high @claude adapt

Critical: Blocked

Cannot adapt breaking changes, or plan failed. Merge blocked. Requires manual investigation -- @claude provides guided analysis.

risk:critical breaking-change
How @claude Interactive Works
The same workflow file handles both automated and interactive triggers

Single Workflow, Two Entry Points

The terraform-consumer-uplift.yml workflow has two trigger sections. Jobs 1-4 run on pull_request events (automated). The interactive job runs on issue_comment when @claude is mentioned. Both share the same checkout, MCP config, and agent context.

When @claude pushes code changes to the PR branch, it re-triggers Jobs 1-4 automatically (via synchronize event). This creates a natural re-validation loop: human asks for help → Claude adapts code → pipeline re-validates → decision updates.

Never Auto-Merge Majors

Major version bumps always require human review, even with zero breaking changes detected.

DESTROY/REPLACE Escalates

Any resource destruction or replacement in the plan escalates to at minimum NEEDS-REVIEW with risk:high.

Conservative Bias

When uncertain, the system recommends needs-review over auto-merge.

Fully Deterministic

No AI involved in the risk assessment. The matrix is applied purely from plan exit code, resource counts, and semver type. AI is reserved for @claude interactive follow-up.

Post-Merge

Apply & Rollback

Separate workflow triggers on merge to main. Includes structured rollback on failure.

Success Path

Upload config tarball → POST /api/v2/runs → poll until applied → confirm run → comment results on merged PR.

applied PR comment

Failure Path (D6)

Comment error details on PR → create incident:apply-failure issue → auto-generate draft rollback PR (git revert) → tag CODEOWNERS.

incident issue rollback PR

Why No Auto-Revert

Auto-reverting merges at enterprise scale is dangerous -- it can cascade through dependent workspaces, trigger unexpected destroys, and bypass change management. The rollback PR goes through the normal pipeline (classify → validate → plan) to ensure the revert is safe before merging. Human control is preserved throughout.
