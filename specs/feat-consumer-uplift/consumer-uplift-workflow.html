USE CASE 1 -- CONSUMER MODULE UPLIFT

Automated Module Version Upgrades for HCP Terraform Consumers

A connected GitHub Actions pipeline that detects module version bumps, validates with Terraform, applies a deterministic risk matrix, and auto-merges safe changes -- with @claude interactive escalation for complex upgrades.

DETECT

Dependabot scans the private registry monthly. A fallback cron scanner catches submodule paths Dependabot misses. Both create PRs in the same format.

ASSESS

A deterministic risk matrix classifies changes by semver type, plan exit code, and destroy/replace presence. No AI needed -- purely scripted.

DECIDE

Structured JSON output drives deterministic decisions: auto-merge zero-change plans, request review for any changes, block breaking changes. @claude for follow-up.

Escalation

The @claude Fix Loop

When the pipeline labels a PR as needs-review or breaking-change, comment @claude to have it fix the consumer code and push -- the pipeline then re-runs automatically

Low Risk: Auto-Merge

No adds, no changes. Plan shows zero infrastructure changes (exit 0). Pipeline auto-merges with squash. No human intervention needed.

risk:low auto-merge

This is the only auto-merge path. The module version bump was applied but resulted in zero infrastructure changes -- no new resources added and no existing resources modified. This applies equally to patch, minor, and major bumps.

  • Plan exit code: 0 (success, no changes)
  • Action: squash merge + risk:low label
  • No reviewer requested

Low Risk: Adds Only (Needs Review)

New resources added but no existing resources modified. Low risk since nothing existing is touched, but requires review to confirm new resources are expected.

risk:low needs-review

New resources will be created but no existing resources are changed, destroyed, or replaced. This is low risk for patch/minor bumps. For major bumps, risk escalates to risk:medium because major version changes are more likely to introduce unexpected behavior.

  • Plan exit code: 2 (changes detected, adds only)
  • Action: needs-review label + request reviewer
  • Optional: comment @claude to investigate

Medium Risk: Changes to Existing

Existing resources will be modified (with or without adds). Higher risk because changes affect running workloads. Comment @claude to investigate and reduce blast radius.

risk:medium needs-review

Plan modifies existing resources -- this affects running workloads and carries higher risk than adds-only. For patch/minor bumps this is risk:medium; for major bumps it escalates to risk:high.

  • Plan exit code: 2 (changes to existing resources)
  • Action: needs-review label + request reviewer
  • Recommended: comment @claude to investigate

High Risk: DESTROY/REPLACE

Plan contains resource destruction or replacement. Always classified as breaking-change regardless of semver type. Merge blocked. Comment @claude to fix consumer code.

risk:high @claude fix

DESTROY/REPLACE actions are always treated as breaking changes, regardless of semver type. For patch and minor bumps this is risk:high; for major bumps it escalates to risk:critical.

  • Plan contains destroy or replace actions
  • Action: breaking-change label + merge blocked
  • Recommended: comment @claude to adapt config

Critical: Plan Failed

Plan failed (exit 1). The module version bump introduces changes that prevent Terraform from planning. Merge blocked. Comment @claude to diagnose and fix.

risk:critical breaking-change

Plan failure means the module interface changed in a way that breaks the consumer code. For patch and minor bumps this is risk:high; for major bumps it escalates to risk:critical.

  • Plan exit code: 1 (error)
  • Action: breaking-change label + merge blocked
  • Comment @claude to diagnose, fix, and push
  • Pipeline re-validates automatically after fix
How @claude Interactive Works
Two workflow files coordinate: the uplift pipeline posts a review comment, and the Claude Review workflow handles the interactive fix loop

Two Workflows, One Pipeline

The pipeline uses two separate workflow files. terraform-consumer-uplift.yml handles the deterministic pipeline (Jobs 1-4) on pull_request events. terraform-claude-review.yml handles interactive @claude review on issue_comment events.

Workflow 1: terraform-consumer-uplift.yml runs the four deterministic jobs (Classify, Validate, Risk Assess, Decision) on PR opened and synchronize events. The Decision job (Job 4) auto-posts an @claude review comment on the PR when the outcome is needs-review or breaking-change.

Workflow 2: terraform-claude-review.yml triggers on issue_comment events when @claude is mentioned. It includes a gate job that prevents bot comments (including its own) from cancelling in-progress reviews. Claude checks out the PR branch, uses MCP tools to investigate module interface changes, fixes consumer code, and pushes.

When @claude pushes code fixes, the synchronize event re-triggers the uplift pipeline (Workflow 1) automatically. This creates a feedback loop:

  • Decision job (Workflow 1) posts @claude review comment on the PR
  • Claude Review workflow (Workflow 2) triggers on the comment
  • Gate job filters out bot comments to prevent cancellation loops
  • Claude diagnoses, fixes consumer code, and pushes
  • Push triggers synchronize event on the PR
  • Uplift pipeline (Workflow 1) re-runs Jobs 1-4 with the fixed code
  • Risk re-assessed -- may now auto-merge if plan is clean

Claude never approves or merges -- the pipeline handles that. The two-file architecture ensures issue_comment events only trigger the lightweight Claude review, not the full 4-job pipeline.

Adds vs Changes

The matrix distinguishes between new resources (adds) and modifications to existing resources (changes). Adds-only is lower risk than changes.

Adding new resources doesn't touch existing infrastructure, so it's classified as low risk with needs-review. Modifying existing resources carries higher risk (medium) because it can affect running workloads. Both always require review -- only zero-change plans auto-merge.

DESTROY/REPLACE Always Blocks

Any resource destruction or replacement always results in a breaking-change classification and merge block.

Resource deletion and replacement are irreversible operations. DESTROY/REPLACE is never classified as merely "needs-review" -- it always blocks the merge as a breaking change. For major bumps, it escalates to risk:critical.

Major Bumps Escalate

Major version bumps escalate the risk level by one tier compared to patch/minor for the same plan outcome.

A major version signals potential breaking API changes. Adds-only goes from low to medium, changes go from medium to high, and DESTROY/REPLACE goes from high to critical. Zero-change plans remain low risk even on major bumps.

Fully Deterministic

No AI involved in the risk assessment. The matrix is applied from plan add/change/destroy counts and semver type.

AI is reserved for @claude interactive follow-up only. The risk assessment uses a simple lookup table with five conditions checked in priority order. This ensures consistent, auditable decisions across all module upgrades.
